# Архитектура

## Цель

Локально (на Mac, из Cursor) превращать PDF в **редактируемый** DOCX,
максимально сохраняя исходный вид. Папка `inputs/` → папка `outputs/`.

## Главный компромисс

PDF — формат с **абсолютным позиционированием** (у каждого глифа свои
координаты). DOCX — формат с **потоковой вёрсткой** (абзацы/таблицы перетекают).
Эти модели несовместимы по своей природе, поэтому:

> 100% визуальной точности **и** 100% редактируемости одновременно недостижимы
> ни одним инструментом. Чем сложнее дизайн исходника, тем сильнее компромисс.

Мы целимся в **гибрид**: текст обычным редактируемым потоком, абсолютные фреймы —
только для вёрстки, которую иначе не передать.

## Конвейер

```
            ┌─────────────┐
  PDF  ───▶ │  analyzer   │  цифровой / скан? сколько текста? шифрование?
            └──────┬──────┘
                   ▼
            ┌─────────────┐   --engine local  (по умолчанию, офлайн)
            │   engine    │   --engine adobe  (облако, выше точность)
            └──────┬──────┘
                   ▼
                 DOCX  ───▶  outputs/
```

| Модуль | Ответственность |
|---|---|
| `convert.py` | CLI, батч `inputs/→outputs/`, выбор движка, логи, коды выхода |
| `pdf2docx_converter/analyzer.py` | Pre-flight: цифровой PDF vs скан (PyMuPDF) |
| `pdf2docx_converter/engine_local.py` | Офлайн-движок на `pdf2docx` |
| `pdf2docx_converter/engine_adobe.py` | Облачный движок на Adobe PDF Services |
| `pdf2docx_converter/config.py` | Папки I/O, параметры качества |

## Движки

### `local` — pdf2docx (по умолчанию)

- Полностью офлайн, бесплатно, без лимитов.
- Реконструирует поток Word: абзацы, простые таблицы, картинки, базовые стили.
- **Сильные стороны:** текстовые документы (письма, договоры, отчёты).
- **Слабые стороны:** дизайнерская вёрстка (инфографика, чарты из векторных
  фигур), нестандартные шрифты. Подробнее — раздел «Ограничения».

### `adobe` — Adobe PDF Services (Export PDF)

- Облачный (файл загружается в Adobe), бесплатный тариф ~500 конвертаций/мес.
- Заметно лучше держит сложную вёрстку и Type3-шрифты.
- Опциональная зависимость `pdfservices-sdk` (`requirements-adobe.txt`).
- Ключи: env / `.env` / `pdfservices-api-credentials.json` (см. README).
- **Не использовать** для документов, которые не должны покидать машину.

## Ограничения (важно)

На практике подтверждено на реальном дизайнерском PDF (бизнес-инфографика,
экспорт из дизайн/веб-инструмента):

1. **Векторная графика ≠ объекты Word.** Бар-чарты, карточки, иконки — это сотни
   векторных фигур. У Word нет нативного редактируемого эквивалента; локальный
   движок превращает их в кашу из картинок и вложенных таблиц.
2. **Type3-шрифты ломают извлечение текста в pdf2docx** — слова склеиваются
   (теряются пробелы). При этом сам PyMuPDF извлекает текст таких PDF корректно,
   что оставляет пространство для собственного текстового движка в будущем.

Вывод: для дизайнерских PDF используем `--engine adobe`; для текстовых — `local`.

## Журнал ключевых решений

| Решение | Почему |
|---|---|
| Python + pdf2docx как база | Лучший баланс «редактируемо + похоже» в опенсорсе |
| Гибридный режим, а не «всё во фреймы» | Сохранить редактируемость |
| Батч `inputs/→outputs/` вместо GUI | Запуск из Cursor одним скриптом (запрос пользователя) |
| OCR отложен | Нужны только цифровые PDF на старте |
| Добавлен Adobe-движок | Локальный движок не тянет дизайнерскую вёрстку/Type3 |
