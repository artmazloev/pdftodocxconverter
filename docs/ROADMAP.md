# Roadmap v2 — план развития (хендофф для новой сессии)

Этот документ — точка входа для продолжения разработки. Core готов; ниже два
запланированных направления, их архитектура, решения и разбивка на задачи (issues).

**Статус:** спланировано, не реализовано. Реализуем в следующих сессиях.

## Что уже готово (контекст)

PDF/HTML → редактируемый DOCX. Движки `local` (`pdf2docx`, офлайн) и `adobe`
(Adobe PDF Services, облако). Для HTML: фикс шрифтов (вариативные→статические,
инжект встроенного Manrope из `assets/`, `--clean-svg-text`) и рендер чистого PDF
через headless Chrome. Постобработка DOCX. Подробно — [ARCHITECTURE.md](ARCHITECTURE.md),
[HTML_TO_DOCX.md](HTML_TO_DOCX.md), [FONTS.md](FONTS.md).

Текущий конвейер `HTML → DOCX`:
```
HTML → [html_prep: шрифты] → [Chrome: чистый PDF] → [Adobe/local] → [postprocess] → DOCX
```

## Два направления

### Эпик 1 — Вывод в PowerPoint (PPTX) через Adobe (issue #24)

**Зачем.** Модель PowerPoint (фиксированный холст + абсолютное позиционирование)
ближе к PDF, чем потоковый Word. Для дизайнерских макетов это даёт лучший
результат: фигуры/кружки — нативные shapes (уходит артефакт «кружок → ромб»),
макет не «расплывается». Минус: PPTX печатается хуже (масштаб-под-лист, поля) —
печать делать через экспорт в PDF. Word остаётся лучшим выбором, когда нужна
надёжная печать редактируемого файла.

**Ключевая идея архитектуры.** Формат вывода **ортогонален** движку и пайплайну.
Этапы подготовки HTML и рендера чистого PDF **переиспользуются 1:1**. Меняется
только: (а) target у Adobe, (б) расширение файла, (в) постобработка.

```
HTML → [html_prep] → [Chrome PDF] → [Adobe: target=DOCX|PPTX] → [postprocess?] → out.(docx|pptx)
            ОБЩЕЕ                      ← здесь развилка формата →
```

- Adobe Export PDF поддерживает PPTX нативно: `ExportPDFTargetFormat.PPTX`.
- `pdf2docx` (движок `local`) умеет **только DOCX**.

**Матрица возможностей (движок × формат):**

| | docx | pptx |
|---|---|---|
| `local` | ✅ | ❌ |
| `adobe` | ✅ | ✅ |

`local + pptx` → понятная ошибка «PPTX доступен только через `--engine adobe`».

**Швы (что трогаем):**
- `engine_adobe.convert_file()` — параметр `target_format`.
- `core`/`convert.py` — ось `--format {docx,pptx}`, расширение из формата, валидация по матрице.
- `postprocess.py` — формат-зависимость: PPTX это другой OOXML (`ppt/slides/slideN.xml`,
  не `word/document.xml`), поэтому для pptx на старте постобработку **пропускаем**.

**Задачи:** #14 (фундамент) → #15 (ось формата) → #16 (Adobe target PPTX) →
#17 (postprocess формат-зависимый) → #18 (доки + smoke).

### Эпик 2 — Десктоп-приложение для macOS (.app) (issue #25)

**Зачем.** Запуск двойным кликом, выбор опций мышью, работа «из коробки» без venv.

**Ключевая идея архитектуры.** GUI — **тонкая обёртка над тем же ядром**, что и
CLI. Никакой логики конвертации в UI. Контролы маппятся на параметры `core.convert`:
- движок: Локально / Adobe (облако)
- формат: Word (.docx) / PowerPoint (.pptx) — **PPTX активен только при Adobe**
  (та же матрица возможностей из эпика 1)
- чекбокс «Чистить подписи на схемах» (`--clean-svg-text`)

**Главные риски/решения (см. spike #19):**
- **Chromium для Playwright** — самое сложное в бандле: вложить браузер в .app
  или докачивать при первом запуске; путь через `PLAYWRIGHT_BROWSERS_PATH`.
- **Ключи Adobe** — в упакованном приложении нет `.env` в репо. Хранить в
  пользовательском конфиге (`~/Library/Application Support/<app>/` или Keychain).
- **assets/Manrope** — включить в data files упаковщика.
- Упаковщик (py2app/PyInstaller/briefcase) и GUI-тулкит (Tkinter/PyQt/rumps) —
  решаются в spike #19.

**Задачи:** #14 (фундамент) → #19 (spike) → #20 (GUI) / #21 (бандл) → #22 (ключи) → #23 (сборка .app/.dmg).

## Общий фундамент (issue #14) — делать первым

Вынести оркестрацию из CLI в публичную функцию ядра:

```
pdf2docx_converter/core.py:
    convert(input_path, *, engine, output_format, clean_svg_text,
            font, postprocess, pages, overwrite) -> Path
```

`convert.py` (CLI) и будущий GUI — тонкие обёртки над `core.convert`. Каждый
движок объявляет `supported_formats`; ядро валидирует комбинацию «движок × формат».
Это разблокирует оба эпика (PPTX = новый формат; .app = новый фронт).
